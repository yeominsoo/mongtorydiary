# 2026-03-19 Handoff

## 오늘 정리한 내용
- `AGENTS.md`와 `docs/planning.md`를 기준으로 웹과 앱을 모두 지원하는 개발 방향을 재정리했다.
- 위젯 지원 요구사항이 추가되면서, 단순 웹 중심 구조보다 멀티클라이언트 구조가 필요하다는 점을 명확히 했다.
- 비교 검토 끝에 앱 메인 프론트엔드는 Flutter로 가는 쪽이 적합하다고 정리했다.
- 새 문서와 갱신 문서로 Flutter 기준 구조를 기록했다.
- `AGENTS.md`를 Flutter 중심 운영 규칙과 handoff 즉시 기록 규칙으로 갱신했다.
- `mobile-flutter/` 기본 디렉토리 초안을 생성했다.
- 저장소 내부 `.tooling/flutter-sdk`에 Flutter SDK를 설치했다.
- `mobile-flutter/`를 실제 Flutter 프로젝트로 초기화하고 `android/`, `ios/` 플랫폼 디렉토리를 생성했다.
- 기본 생성 앱을 Mongtory Diary 시작 화면으로 교체했다.
- Flutter 앱 진입점을 `app`, `core/theme`, `presentation/screens`로 분리해 최소 계층 구조 정리를 시작했다.
- `pub.dev` 접근 실패 원인이 현재 세션의 네트워크 제한임을 확인했다.
- `startup -> sign-in -> home shell` 기준의 MVP 라우팅 골격을 추가했다.
- 홈 셸에는 일기, 캘린더, 몽토리 탭의 기본 화면 구조를 배치했다.
- 사용자로부터 관리자 권한 작업에 대한 포괄 승인을 받았고, 비밀번호 자체는 문서에 남기지 않기로 정리했다.
- `flutter analyze` 결과에 맞춰 `main.dart` import 누락과 기본 위젯 테스트 참조를 수정했다.
- `flutter analyze`에서 남은 unused import 경고 1건을 정리했다.
- 로컬 검증 결과 `flutter analyze`는 `no issues found!`, `flutter test`는 `All tests passed!` 상태를 확인했다.
- 홈 셸 내부 탭 화면을 `diary`, `calendar`, `profile` 개별 파일로 분리했다.
- 공통 카드 UI를 `presentation/widgets/section_card.dart`로 추출했다.
- `flutter_riverpod` 의존성을 추가하고 `ProviderScope`를 앱 진입점에 연결했다.
- 세션 상태와 홈 탭 상태를 Riverpod provider로 분리했다.
- 로그인 화면과 홈 셸 화면이 Riverpod 상태를 사용하도록 갱신했다.
- Riverpod 추가 이후 사용자 로컬 실행 기준으로 `flutter pub get`, `flutter analyze`, `flutter test`가 모두 정상 통과했다.
- `User`, `Diary`, `Emotion`, `CalendarSummary` 기준의 API/DTO 초안을 `docs/api-spec.md`로 정리했다.
- 백엔드에 `ApiResponse`, 인증/감정/일기/캘린더 DTO, 서비스, 컨트롤러 mock 골격을 추가했다.
- `AuthControllerTest`, `DiaryControllerTest`로 기본 응답 래핑 동작을 검증하는 테스트 골격을 추가했다.
- 이 환경에서 `mvn test`는 `repo.maven.apache.org` DNS 해석 실패로 검증하지 못했다.
- Flutter `domain/models`, `data/dto`, `data/mappers`를 `docs/api-spec.md` 기준 필드 구조로 정리했다.
- 세션 상태가 단순 문자열이 아니라 `AuthSession` 도메인 모델을 직접 참조하도록 변경했다.
- Flutter에 `domain/repositories`, `data/datasources/mock`, `data/repositories`를 추가해 mock 데이터 계층을 연결했다.
- 로그인, 일기, 캘린더, 감정 타입 화면 상태가 repository 기반 mock 데이터 흐름을 사용하도록 갱신했다.
- 새 세션에서 컨텍스트가 끊겨도 바로 이어받을 수 있도록 `docs/session-continuity.md` 상세 인수인계 문서를 추가했다.
- 백엔드에 `DiaryEntry` 엔티티와 `DiaryEntryRepository`를 추가하고 Diary/Calendar 서비스를 SQLite 영속화 방향으로 전환했다.
- 기본 일기 1건을 넣는 초기화 컴포넌트를 추가해 테스트 가능 환경에서 초기 응답이 비어 있지 않도록 정리했다.
- Maven 테스트는 저장소 내부 `.m2` 캐시로 재시도했지만 `repo.maven.apache.org` DNS 해석 실패로 끝까지 검증하지 못했다.
- Flutter 앱이 `DATA_SOURCE_MODE`, `API_BASE_URL` `dart-define`으로 `mock/remote`를 전환할 수 있게 정리했다.
- 로그인 화면에 세션 로딩/실패 상태를 추가하고, 일기/캘린더 화면이 현재 데이터 소스와 API 오류를 표시하도록 갱신했다.
- 백엔드에 `UserAccount` 엔티티와 `UserAccountRepository`를 추가하고 회원가입/로그인/토큰 재발급이 SQLite 기반 계정 데이터로 동작하도록 `AuthService`를 전환했다.
- 기본 사용자 계정을 넣는 초기화 컴포넌트와 SHA-256 기반 비밀번호 해시 서비스를 추가했다.
- Flutter 일기/캘린더/프로필 화면이 단순 건수 표시를 넘어서 실제 응답 항목 일부를 렌더링하도록 보강했다.
- 백엔드가 access token으로 현재 사용자를 식별하고 Diary/Calendar를 사용자 소유 데이터 기준으로 조회하도록 전환했다.
- Flutter remote datasource가 세션 access token을 `Authorization` 헤더로 붙여 Diary/Calendar API를 호출하도록 갱신했다.

## 이번 추가 작업 후 동작 상태
- 새 세션에서는 `AGENTS.md -> docs/project-status.md -> docs/session-continuity.md -> docs/handoff/2026-03-19.md` 순서로 읽으면 현재 프로젝트 컨텍스트를 빠르게 복구할 수 있다.
- Diary API와 Calendar API는 이제 mock 고정 응답이 아니라 `DiaryEntry` 저장 데이터를 기반으로 응답하도록 구조가 바뀌었다.
- 초기 실행 시 저장된 일기가 없으면 기본 샘플 일기 1건이 주입된다.
- Flutter는 기본적으로 mock 모드지만, `dart-define`으로 remote 모드 실행이 가능하다.
- 로그인 실패와 API 조회 실패가 기본 화면에서 바로 드러나도록 상태와 문구가 보강되었다.
- Auth API는 이제 시드 사용자와 신규 가입 사용자 데이터를 SQLite에 저장하고 읽는 구조가 되었다.
- Flutter 화면에서 remote 모드 시 일기 제목, 날짜, 감정 코드, 캘린더 일자별 건수, 사용자 이메일, 감정 타입 목록 등을 바로 확인할 수 있다.
- Diary/Calendar API는 이제 로그인으로 발급받은 access token 기준 현재 사용자 데이터만 반환한다.
