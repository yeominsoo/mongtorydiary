# Mongtory Diary 플랫폼 개발 방향

## 목표
Mongtory Diary는 감정형 일기 앱을 중심으로 성장하고, 별도 웹 프론트엔드와 안드로이드/iOS 위젯까지 확장하는 구조를 목표로 한다. 이 기준에서는 모바일 앱 본체의 완성도와 위젯 연동성이 가장 중요하다.

## 최종 방향
프론트엔드 전략은 `Flutter 앱 + 별도 웹 FE`로 확정하는 것이 가장 적절하다. 현재 `frontend/`는 초기 골격 수준이므로, 지금 전환하면 손실이 작고 장기 구조는 더 안정적이다.

- 앱 메인: Flutter
- 웹: 별도 FE 프로젝트
- 백엔드: Spring Boot API
- 위젯: Android App Widget 또는 Glance, iOS WidgetKit

웹과 앱 UI를 억지로 하나의 코드베이스로 묶지 않고, 공통은 API 계약과 도메인 규칙 수준에서 맞춘다.

## Flutter를 선택하는 이유
- 감성적인 캐릭터 UI와 애니메이션 표현력이 높다.
- 모바일 앱 중심 서비스에서 화면 일관성과 확장성이 좋다.
- 위젯은 결국 네이티브 확장이 필요하므로, React 기반 통합의 이점이 이번 프로젝트에서는 작다.
- 웹을 새로 만들 계획이라면 기존 React 자산을 유지해야 할 이유가 줄어든다.

## 위젯 지원 원칙
위젯은 앱의 축소판이 아니라 "오늘 상태를 빠르게 확인하고 앱으로 진입하는 요약 진입점"으로 설계한다.

- 오늘 일기 작성 여부
- 오늘 감정 스티커
- 최근 연속 작성 일수
- Mongtory 한 줄 메시지
- 앱 특정 화면으로 이동하는 딥링크

복잡한 편집은 앱에서 처리하고, 위젯은 조회와 빠른 진입만 담당한다.

## 백엔드 우선 설계
Flutter 앱, 웹, 위젯이 같은 데이터를 소비하려면 백엔드를 먼저 API 중심으로 고정해야 한다.

- 인증: 회원가입, 로그인, 토큰 재발급
- 일기: CRUD, 날짜별 조회, 월간 캘린더 조회
- 감정: 감정 스티커 목록, 일기별 감정 저장
- 이미지: 업로드 메타데이터와 접근 경로
- 동기화: 수정 시각과 클라이언트 동기화 기준

응답은 `ApiResponse<T>`로 통일하고, Entity 직접 노출 없이 DTO로 통신한다.

## 단계별 실행 순서
### 1단계: API MVP
- 인증 API
- 일기 CRUD API
- 월간 캘린더 API
- 감정 스티커 API

### 2단계: Flutter 앱 MVP
- 로그인
- 일기 목록, 작성, 상세, 수정
- 월간 캘린더
- 감정 선택
- 로컬 캐시와 기본 동기화

### 3단계: 위젯
- 안드로이드 홈 위젯
- iOS WidgetKit 위젯
- 위젯 전용 요약 데이터와 딥링크 연결

### 4단계: 별도 웹 FE
- 소개/랜딩
- 계정 기반 웹 사용 시나리오
- 필요 시 관리자 또는 백오피스 화면

## 결론
이 프로젝트는 웹과 앱을 동시에 키우는 구조보다, Flutter 앱을 메인 제품으로 두고 웹을 별도 채널로 분리하는 편이 더 적합하다. 위젯까지 고려하면 Flutter 기준의 앱 중심 구조가 장기 확장성과 제품 완성도 측면에서 유리하다.
