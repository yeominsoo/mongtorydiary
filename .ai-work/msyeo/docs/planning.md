# Mongtory Diary 제품 계획

## 비전
Mongtory Diary는 캐릭터 기반 감정형 일기 서비스다. 사용자가 매일의 감정과 기록을 부담 없이 남기고, Mongtory 캐릭터와의 상호작용을 통해 꾸준히 돌아오고 싶은 일기 경험을 만드는 것이 목표다.

## 현재 제품 방향
- 메인 제품: Flutter 모바일 앱
- 백엔드: Spring Boot API 서버
- 웹: 기존 `frontend/`는 초기 React 프로토타입으로 보존하고, 정식 웹은 필요 시 별도 FE로 분리
- 위젯: Android App Widget 또는 Glance, iOS WidgetKit
- 문서 루트: `.ai-work/msyeo/docs/`

## MVP 범위
### 백엔드
- 회원가입, 로그인, 토큰 재발급
- 일기 목록, 상세, 생성, 수정, 삭제
- 월간 캘린더 요약 조회
- 감정 타입 목록 조회
- `ApiResponse<T>` 공통 응답

현재 백엔드는 위 범위의 골격을 구현했다. 다만 인증은 JWT가 아니라 DB에 저장하는 임시 access/refresh token 방식이며, 전역 예외 응답 표준화와 Spring Security 구성은 아직 없다.

### Flutter 앱
- 시작 화면
- 임시 로그인 화면
- 홈 셸 탭: 일기, 캘린더, 몽토리/프로필
- mock/remote 데이터 소스 전환
- remote 모드에서 로그인 후 일기/캘린더 API에 `Authorization` 헤더 전달

현재 Flutter 앱은 목록과 요약 표시 중심의 MVP 골격이다. 실제 작성/상세/수정 폼, 회원가입, 토큰 재발급, 영구 세션 저장, 위젯 연동은 아직 없다.

## 2단계 후보
- Flutter 로그인 화면을 실제 입력 폼으로 확장
- 일기 작성/상세/수정 화면 구현
- API 오류와 로딩 상태를 화면별로 구체화
- 백엔드 인증 토큰 정책 고도화
- 사용자 소유권 검증 테스트 보강
- API 계약과 Flutter DTO 매핑 정합성 유지

## 3단계 후보
- 이미지 업로드 또는 이미지 URL 관리 정책
- 감정별 통계와 검색/필터
- 연속 작성 일수와 보상/업적 시스템
- 위젯용 요약 API 또는 앱 로컬 캐시 설계
- Android/iOS 위젯 구현

## UX 방향
- 모바일 앱 중심의 단순하고 따뜻한 기록 흐름
- 위젯은 편집 UI가 아니라 오늘 상태 요약과 앱 진입점 역할
- 캐릭터 반응, 감정 스티커, 연속 작성 보상은 MVP 이후 점진 도입

## 현재 작업 준비 기준
핸드오프 기준으로 다음 작업은 API 계약 정합성 정리 이후 아래 순서가 자연스럽다.
1. 백엔드와 Flutter remote 모드의 실제 연결 검증
2. Flutter 로그인 입력 폼과 세션 상태 고도화
3. 일기 작성/상세/수정 플로우 구현
4. 백엔드 인증/소유권 테스트 보강
5. 위젯용 요약 데이터와 딥링크 규칙 문서화
